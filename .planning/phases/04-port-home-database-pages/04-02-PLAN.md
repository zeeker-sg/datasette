## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Caddy | Static asset requests (`*.css`, `*.woff2`) arrive over untrusted network. |
| Caddy → frontend StaticFiles | Suffix-matched static paths forwarded; no dynamic behavior. |
| StaticFiles → local filesystem | Starlette serves files from the mounted directory only; path traversal rejected by StaticFiles. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Tampering | zeeker.css contents (supply-chain via accidental edit) | mitigate | CSS is sed-extracted from committed M1 source; grep-based sanity checks in verify step catch unbalanced braces, missing `@font-face` count, and missing expected selectors before the commit lands. |
| T-04-02-02 | Information Disclosure | Font files served under /static/fonts/ | accept | Public fonts already self-hosted in M1; no user data. Re-serving from frontend has identical disclosure profile. |
| T-04-02-03 | Denial of Service | Large CSS payload | mitigate | Target ~2,300 lines (~60KB); browser-cacheable. Well below DoS thresholds. If a harvest accidentally includes all 4,116 lines, the size check (≤3,500 lines) fails the verifier. |
| T-04-02-04 | Tampering | Font file corruption during copy | mitigate | `diff` on file sizes vs source ensures byte-identical copy. A corrupt woff2 would render fallback system fonts — visually detectable at the Plan 04-05 checkpoint. |
| T-04-02-05 | Elevation of Privilege (cache staleness on deploy) | Browser-cached old CSS after deploy | accept | Per CONTEXT, cache-busting is explicitly deferred. `max-age=60` header on HTML (Plan 04-03/04) means worst-case 60s of stale CSS on first visit after deploy; hard-refresh resolves. Documented in Plan 04-05's human-checkpoint notes. |
