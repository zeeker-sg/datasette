Additional rules (carried forward from skill):

- Italic-accent `<em>` on every H1 — `font-style: italic; color: var(--color-accent); font-weight: 500;`

**Exception clause — locked design-system override:**

bespoke per-phase scales; this project's locked design system (sketch-findings) supersedes it.

each serving non-overlapping content. Phase 5 already established this exception clause; Phase 6
inherits it without re-deriving.

## Page Surfaces and Layout

Every aux-page route gets a body class (`page-developers`, `page-status`, `page-sources`,
`page-about`, `page-how-to-use`, `page-search`, `page-sql`) so the Phase-6 CSS can scope its
selectors and avoid leaking into Phase 4-5 surfaces. The body class is set by passing
`page_class="page-{slug}"` from the route handler into the `base.html` Jinja context; `base.html`
gets a one-line edit to bind it to `<body class="{{ page_class or '' }}">`. (Planner picks
exact mechanism; the contract is "scoping is done via body class.")

### Common shell (all aux routes, inherited from `base.html`)

- Dark editorial nav (`.db-nav`)
- Breadcrumb strip — for aux pages: `Home › {Page Name}`. The current page label is the H1 plain
  text. (No deeper nesting on aux routes except `/sql/{db}` which is `Home › SQL › {db}`.)
- Page hero — italic-accent H1 + lede subtitle. Aux pages re-use `.guide-hero` from M1
  (full-bleed warm-paper card with petrol rule-top, centered layout) — adapted slightly to fit
  the broadsheet shell. Detail in CSS Harvest below.
- 4-column paper footer (already in `base.html`)
- **No `.db-statband` (petrol stat band) on aux pages** EXCEPT `/status`, which renders its own
  stats-simple band as a `.aux-card` block, not a full-bleed petrol band. (Per skill: stat band
  reserved for home / database / table.)

### `/search` — Cross-database FTS UI (NEW)

This is one of two new surfaces. No prior sketch session — design derives from skill rules + Phase 5 feed-card pattern.

Layout: search-form region + results region, max-width 960px, centered. Two states.

#### State A — empty query (`?q=` not set or empty)

```
[hero card with petrol rule-top]
  Kicker:    № 01 · SEARCH
  H1:        Search across <em>everything</em>
  Lede:      Look across every searchable table on data.zeeker.sg.
            Results group by database and table, ranked by relevance.

[search form, full-width inside hero card]
  Inline:    [text input, placeholder="Search FTS-indexed tables..."] [Submit button]

[search-tips card] (below hero)
  H2:        Search tips
  ul:
    - Use quotes for exact phrases: "data centre"
    - Use AND / OR / NOT (FTS5 operators)
    - Wildcard suffixes: invest* matches investment / investor
```

No "recent searches" — D-06 + Discretion default. No telemetry.

#### State B — query submitted (`?q={term}` set)

```
[hero card]
  H1:        Results for <em>{term|escape}</em>
  Search form (re-rendered with current value pre-filled)
  Lede meta: "{total_count} results across {N_databases} databases · {elapsed_ms}ms"

[results region]
  if total == 0:
    .search-empty card:
      H2: No results for "{term}"
      Lede: "Try broadening your search, removing operators, or checking for typos."
      ul of search tips (same as State A)
  else:
    For each (db, table) group with results.length > 0, in alphabetical order:
      <section class="search-group">
        <header class="search-group-head">
          <h2><a href="/{db}/{table}?_search={term}">{db_title} › {table_title}</a></h2>
          <span class="result-count">{count} {result|pluralize}</span>
        </header>
        <ol class="search-results">
          [up to 10 .va-item-style result cards — see anatomy below]
        </ol>
        if count > 10:
          <a class="see-all" href="/{db}/{table}?_search={term}">See all {count} results in {table_title} →</a>
      </section>
```

Result card anatomy (one row per FTS hit) — based on Phase 5 `.va-item`:

- Container: `<li class="search-result">`
- Top row: `.search-result-meta` — kicker mono terracotta uppercase ("RESULT №01" or column-name kicker)
- Title: `<h3 class="search-result-title">` Fraunces 500 — links to row detail page (`/{db}/{table}/{pk}`) when PK derivable; otherwise to the table page filtered by FTS.
- Excerpt: `<p class="search-result-excerpt">` Inter sm — first 240 chars of the highest-priority text column with `<mark>` highlights on FTS match terms.
- Footer: `.search-result-foot` — mono xs muted: `{db_name} · {pk_or_id} · {date_field|safe_format if any}`

`<mark>` styling: `background: rgba(192,138,46,0.25); color: var(--color-ink); padding: 0 2px; border-radius: 2px;` (Phase 5 pattern).

Empty-table case (a group has 0 results): omit the group entirely. Don't show "0 results in X."

#### Loading / partial-failure state

If one or more per-table fan-out requests fail (timeout, 5xx), still render the groups that succeeded. Append a single line at the bottom of the results region:

> "Search timed out for {N} table(s). [Retry →](?q={term}&_retry=1)"

(Discretion: planner may simplify to silently dropping failed groups for v1; if so, log a warning server-side. UI-checker accepts either behavior as long as the page does not 500.)

#### Form mechanics

- Method: `GET`, action `/search`, single `<input name="q" type="search">` + submit `<button>` "Search".
- The form is `autocomplete="off"`; placeholder text "Search FTS-indexed tables…".
- After submit, the input retains its value (`<input value="{{ q|e }}">`).
- The form sits inside the hero card on both states. On State B, an additional small `[× clear]` link beside the submit button (or an `aria-label="Clear search"` clickable) navigates to `/search` (drops `?q`).

### `/sql` and `/sql/{db}` — SQL editor (NEW)

Two URLs, two surfaces.

#### `/sql` — landing list

Layout: small page, max-width 720px, centered.

Sections:

1. **Hero** — `.guide-hero` with H1 "Run <em>SQL</em>" + lede ("Pick a database to write a query against.").
2. **Database list** card — `<ul class="sql-db-list">` of databases. Each `<li>` is a clickable card-row: `<a href="/sql/{db}">` + database title-cased + table count + (optional) database size in filesizeformat. Empty state: "No databases available." (mirrors `/sources`).
3. **Footer note** small text — link to `/developers` for API docs.

`.sql-db-list` styling: editorial-row pattern (carried from Phase 4 database list) — full-width row, `.row:hover` left-edge slider bar in petrol, mono table-count on right.

#### `/sql/{db}` — editor

Layout: editor at top, results below. Max-width 1024px, centered.

Sections:

1. **Hero** — `.guide-hero` (compact, smaller padding than other aux pages — `--space-8` not `--space-16`) with H1 "SQL · <em>{db_title}</em>" + lede ("Read-only queries against {db}. Time limit: 3 seconds. Row cap: 1000.").
2. **Saved queries (canned)** — only when `metadata.json` defines `databases.{db}.queries.*`. Renders as a `<details>` (default-collapsed) with `<summary>` "Saved queries ({count})". Inside, an unstyled `<ul>` of `<button class="canned-query" type="button" data-sql="{q.sql|e}">{name}</button>` items. Each button click pre-populates the textarea (small JS, ~5 lines, addEventListener). When a canned query has params (`q.params` array OR `:param` placeholders detected), render a sibling input row inline above the textarea.
3. **Editor form** — `<form method="POST" action="/sql/{{ db }}">`:
   - `.sql-param-row` (rendered only when canned query selected and params present): one `<input type="text" name="_sql_param_{name}" placeholder="{name}">` per param.
   - `<textarea name="sql" rows="8" class="sql-textarea">` — JetBrains Mono, `--text-sm`, `min-height: 160px`, full width, `--color-surface` background, 1px `--color-border` border, `--radius-sm` corners.
   - Submit row: left-aligned "Run" button (primary, accent background, paper text); right-aligned "Reset" link (clears textarea via location → `/sql/{db}`).
4. **Results region** — only when handler has executed a query:
   - **Success state:** results header line ("`{rows.length}` rows · `{elapsed_ms}` ms") + scrollable results table (`.sql-results-table`) with column headers from JSON `columns` array + row data. Below the table: **export links row** — `<a href="/{db}.csv?sql={sql|urlencode}">Download CSV ↗</a>` and `<a href="/{db}.json?sql={sql|urlencode}">View JSON ↗</a>`. Both route via Caddy suffix to datasette.
   - **Truncation banner:** if datasette returned the row cap (1000 rows), show a notice line above the export row: "Showing 1,000 rows (the maximum). [Download all matching rows as CSV ↗]." link is the same suffix-routed CSV.
   - **Error state:** `.sql-error` card (terracotta left-border) containing `<pre>{error_message|e}</pre>` + below: "Common causes: typo in column name · table needs square brackets · syntax error".
   - **Cache-Control:** `no-store` on POST per D-14.
5. (Optional, Discretion) **Schema hint sidebar** — a small aside listing the tables in `{db}` with column names. Default: omit. Planner may include if the schema reference list is small and fits in a 240px right column without crowding the editor.

## Interaction Contracts

### `/search` — form behavior

- Form `method="GET"`, action `/search`, single text input + submit. Pressing Enter submits.
- After submit, browser navigates to `/search?q={term}`. Server fans out per `app.state.searchable_tables`.
- Results render in alphabetical (db, table) group order. Within a group, rows are FTS-rank-ordered as datasette returns them; frontend does NOT re-rank cross-group.
- Result-card title link target priority: `/{db}/{table}/{pk}` if PK is derivable from datasette JSON; fall back to `/{db}/{table}?_search={term}` (table page filtered).
- `<mark>` highlight: when datasette returns `_search_highlight` data, apply per Phase 5 pattern. Otherwise the excerpt renders without highlights — DO NOT regex-replace client-side.
- Group anchors: each `.search-group` gets `id="group-{db}-{table}"` so result counts can deep-link.
- Result-region empty: render the no-results card; do not render zero-count groups; do not render the result-meta line.
- Clear: `× Clear` link navigates to `/search` (no `?q`), which renders State A.

### `/sql/{db}` — editor + execution

- Saved-queries disclosure: native `<details>`. JS click handler on `.canned-query` button copies its `data-sql` value into the textarea (`textarea.value = btn.dataset.sql; textarea.focus();`). When a canned query has params, the JS also reveals `.sql-param-row`.
- Form submit: POST to `/sql/{db}` with body `sql={textarea}` + any `_sql_param_*` form fields. Handler URL-encodes them into `/{db}.json?sql=…&_param_*=…` for datasette.
- Re-render: handler returns the `/sql/{db}` template with `sql=<submitted value>`, `results=<datasette response>`, `error=<error string or None>`. Textarea retains the submitted value.
- Reset link: GET to `/sql/{db}` with no query — clears the textarea (handler-side: the template renders empty textarea when `sql is None`).
- Error display: when datasette returns 4xx/5xx with an `error` field, render `.sql-error` block. Do NOT render the results table. Do NOT lose the textarea contents.
- Truncation: when datasette response includes `truncated: true` OR `rows.length === 1000`, render `.sql-truncation` banner with the CSV deep-link.
- Export links: always rendered when results exist. URL-encode `sql` param.
- No client-side syntax highlighting (PRD Appendix B; D-09 keeps deferred).
- No SQL formatting / linting in v1.

### Method-card numbered circle (`.method-number`)

- Visual: 32px diameter circle, `--color-accent` background, paper text, Inter 600 numeric.
- Position: top-left of card, `margin-bottom: --space-3`.
- Copy: digits 1–4 only (matching M1 four-card sequence). When a card has no number (e.g. on `/help` "More Resources"), omit the circle.

### Method-card top-border accent rotation

Match the Phase 4 home card pattern — rotate `border-top-color` per nth-child:

```css
.method-card { border-top: 4px solid var(--color-accent); }
.method-card:nth-child(3n+2) { border-top-color: var(--color-ochre); }
.method-card:nth-child(3n+3) { border-top-color: var(--color-terracotta); }
```

This caps three accent colors per page.

### Hover transitions (all aux pages)

- `.aux-card`: no hover transform (cards are content blocks, not actionable).
- `.method-card`: `border-top-width 0.15s ease` only — no translateY (matches Phase 5 feed cards).
- `.use-case`, `.feature`: `background 0.15s ease` to `--color-bg-alt`.
- `.canned-query` button: `background-color 0.15s ease` to `--color-bg-alt`.
- `.sql-db-list li:hover`: `.row::before` left-edge slider expands (carry-forward Phase 4 pattern).
- `.search-result:hover`: `background-color 0.15s ease` to `--color-bg-alt`. Title link underlines on hover only (`.search-result-title a:hover { text-decoration: underline; }`).
- `.cta-buttons .btn-primary:hover`: `background-color 0.15s ease` to `--color-accent-hover`.

No translateY on aux pages. No drop shadows except the existing card-hover shadow (which only applies to `.method-card`).

### Copy buttons (`.copy-btn` on `.example-box`)

- Visible at top-right of every `.example-box`.
- Click: `navigator.clipboard.writeText(pre.innerText)`. On success, button text temporarily flips to "Copied" + `.copy-btn.copied` class; reset after 1.5s.
- Discretion: planner may implement OR ship the visual button without JS for v1 (text "Copy" non-functional). UI-checker accepts both; auditor will flag if button is missing.

### Page-class scoping

- Every aux-page handler passes `page_class` into the template context. Values: `"page-developers"`, `"page-status"`, `"page-sources"`, `"page-about"`, `"page-how-to-use"`, `"page-search"`, `"page-sql"`, `"page-sql-db"`.
- `base.html` adds: `<body class="{{ page_class or '' }}">`.
- All Phase-6 CSS is scoped under these body classes. Generic classes (`.aux-card`, `.method-card`, etc.) get base styles; page-specific overrides nested under `.page-{slug} .class`.
- `.aux-card` is the ONE class that's globally available (used by every aux page).
