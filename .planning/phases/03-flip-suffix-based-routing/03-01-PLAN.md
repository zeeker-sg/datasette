<objective>
Parameterize the two Phase-2 baseline scripts so Phase 3 (and every later phase)
can target their own baseline directory via env var without editing the script.

Purpose: the Phase-2 baselines lived at `.planning/baselines/phase-02/` —
that directory was renamed to `phase-03-pre/` (commit `ee3f3ad`) so the
hardcoded paths in `scripts/capture_baseline.sh` (line 13) and
`scripts/verify_api_parity.sh` (line 10) are now broken. Rather than re-hardcode
to `phase-03-pre`, parameterize via `ZEEKER_BASELINE_DIR` so Phase 4+ can
re-baseline against `phase-04-pre` etc. without further script edits.

Output: two scripts modified (one line each); smoke-tested by re-running
parity against the current pre-flip stack and confirming the same Phase-2
triage categories (A/B/C/D) appear (because the topology hasn't changed yet —
Wave 2 makes the actual Caddyfile edit).

This plan is the prerequisite for Plan 03's `verify_phase_03.sh`, which exports
`ZEEKER_BASELINE_DIR` before invoking parity, and for Plan 04's `ZEEKER_BASELINE_DIR=.planning/baselines/phase-03-pre bash scripts/verify_api_parity.sh`
verification.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer shell → host filesystem | Script edits write to scripts/*.sh — repo-tracked files |
| script → environment variable | New `ZEEKER_BASELINE_DIR` env var read at script-start; could be set to attacker-controlled path if shell is compromised |
| script → docker / git binaries | Scripts shell out to `git rev-parse` and indirectly to `docker compose`; no new shell-out introduced by this plan |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | scripts/verify_api_parity.sh JQ_STRIP filter | mitigate | The plan's `<action>` blocks both tasks explicitly forbid widening JQ_STRIP. Phase 2's SUMMARY codifies this as the bright line between honest verification and rationalization; this plan inherits that discipline verbatim. The acceptance criteria check that only the BASELINE_DIR line is touched (`git diff` line count is small). |
| T-03-02 | Information disclosure | ZEEKER_BASELINE_DIR env var | accept | The env var contains a filesystem path; it is not secret. Setting it to a sensitive path (e.g., `/etc/`) would cause the script to fail at the `ls "$BASELINE_DIR"/*.json` glob; no read leak occurs. ASVS L1 V14.1.1 — env-var-driven config is acceptable for non-secrets. |
| T-03-03 | Denial of service | parameterized scripts run with attacker-controlled ZEEKER_BASELINE_DIR=/dev/zero or similar | accept | Bash glob `ls /dev/zero/*.json` returns no match → script exits at line 17 with "no baselines found" error. No infinite loop; no resource exhaustion. The test in Task 2's acceptance criteria covers the nonexistent-dir case. |
| T-03-04 | Tampering | scripts/capture_baseline.sh writes a README into OUT_DIR (line 57-71) | accept | Heredoc content is static (no env-var interpolation). If ZEEKER_BASELINE_DIR is set to a directory the user owns, the README write is intended; if set to a directory outside the repo, the user must already have write access to that path → no privilege escalation. ASVS L1 V12.3.1 — file path is constructed but not user-supplied at runtime in this plan's scope. |
| T-03-05 | Repudiation | Smoke-test output in Task 2 is not committed | mitigate | The SUMMARY for Plan 01 records the observed diff categories (or lack thereof). The actual gate run is in Plan 04, which DOES commit forensic logs (per the Phase-2 dual-log capture pattern). Plan 01's smoke is informational; the audit trail begins at Plan 04. |

All five threats: ASVS L1 baseline. None rated `high`. Threat-model gate: PASS.
