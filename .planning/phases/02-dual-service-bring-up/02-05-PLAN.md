Run the bring-up sequence below. Capture every command's output to `.planning/phases/02-dual-service-bring-up/02-05-bringup-log.txt` (this is a transient log committed alongside the SUMMARY for forensics if anything later breaks).

```bash
LOG=.planning/phases/02-dual-service-bring-up/02-05-bringup-log.txt
mkdir -p "$(dirname "$LOG")"
exec > >(tee "$LOG") 2>&1

set -euo pipefail

echo "===== Phase 2 bring-up =====  $(date -u +%FT%TZ)"

echo
echo "--- 1. Tear down any previous attempt (clears orphan containers per RESEARCH Runtime State) ---"
docker compose down --remove-orphans

echo
echo "--- 2. Build (frontend image needs uv sync; datasette image is unchanged) ---"
docker compose build

echo
echo "--- 3. Start in detached mode ---"
docker compose up -d

echo
echo "--- 4. Wait for all services healthy (timeout 120s) ---"
DEADLINE=$((SECONDS + 120))
while [ "$SECONDS" -lt "$DEADLINE" ]; do
  STATES=$(docker compose ps --format json | jq -r '.Health // .State')
  TOTAL=$(echo "$STATES" | wc -l | tr -d ' ')
  HEALTHY=$(echo "$STATES" | grep -c '^healthy$' || true)
  echo "  $(date +%T) — $HEALTHY/$TOTAL healthy"
  if [ "$TOTAL" -eq 3 ] && [ "$HEALTHY" -eq 3 ]; then
    echo "  All three services healthy."
    break
  fi
  sleep 5
done

echo
echo "--- 5. Final compose ps ---"
docker compose ps

echo
echo "--- 6. verify_phase_02.sh ---"
bash scripts/verify_phase_02.sh
```

Expected outcome: `verify_phase_02.sh` exits 0. If it fails, the log file shows exactly which assertion broke. Common failures and fixes:

  - **datasette unhealthy at 60s**: S3 download taking longer than expected, or AWS creds missing. Wait longer; if creds are absent, drop a `.db` into `./data/` (fixtures.db is already there) and the healthcheck still passes via `/-/versions.json` even with no databases.
  - **caddy can't resolve zeeker-datasette**: probably someone added `network_mode:` somewhere; revert and re-up (RESEARCH Pitfall 7).
  - **/frontend-test returns 200 instead of 404 via Caddy**: would mean Caddy is somehow already routing to frontend; check Caddyfile — should NOT have any active `@datasette_api` matcher in Phase 2 (RESEARCH Pattern 2).
  - **frontend healthcheck unhealthy**: probably `python:3.12-slim` vs urllib mismatch; `docker compose exec frontend python -c "import urllib.request; print(urllib.request.urlopen('http://localhost:8000/frontend-test').status)"` should return 200.

If any failure persists after standard fixes, STOP and bring it to the human checkpoint with the failure mode documented.
